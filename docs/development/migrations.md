# Database Migrations

This document outlines the database migration workflow for the url-redir-short project using the Supabase CLI.

## Overview

The `admin-service/supabase/` directory contains the Supabase project configuration.
Database schema modifications must be applied via migrations to ensure deterministic environments across local development, staging, and production.

The original `schema.sql` at `admin-service/supabase/schema.sql` has been preserved as a reference, but the active schema changes should now be captured as iterative migrations in `admin-service/supabase/supabase/migrations/`.

Our migration strategy assumes this is a fresh deployment, meaning our first migration (`20250125000000_baseline.sql`) is a full database representation that builds everything from scratch.

## Helper Script

A wrapper script `scripts/migration.sh` has been provided to simplify common migration tasks:

```bash
# Create a new migration file
./scripts/migration.sh new <name>

# Generate a migration automatically based on schema changes
./scripts/migration.sh generate <name>

# Reset the local database and apply all migrations from scratch
./scripts/migration.sh reset

# Push unapplied migrations to the remote database
./scripts/migration.sh push

# Show the migration history and current status
./scripts/migration.sh status
```

## Creating a New Migration

1. **Manual creation:** Run `./scripts/migration.sh new add_new_feature`.
2. This creates a new file at `admin-service/supabase/supabase/migrations/YYYYMMDDHHMMSS_add_new_feature.sql`.
3. Open the newly created file and write the raw SQL commands required for your change in the `UP MIGRATION` section.

## Rollback Strategy

Every migration file generated by our helper script includes a template with a `DOWN MIGRATION (ROLLBACK)` section at the bottom.

For safety, each migration **should include the corresponding rollback SQL** within the block comment.

```sql
-- ==============================================================================
-- UP MIGRATION
-- ==============================================================================
CREATE TABLE new_table (
  id uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
  name text NOT NULL
);

-- ==============================================================================
-- DOWN MIGRATION (ROLLBACK)
-- ==============================================================================
/*
-- Rollback instructions:
-- Put the SQL to revert the above changes here.

DROP TABLE IF EXISTS new_table;
*/
```

If a deployment goes wrong, you can manually apply the statements in the Down Migration section to reverse the changes. For critical operations (like schema restructuring), having tested rollback SQL is essential.

## Running Migrations Locally

When you make changes to migration files, you can test them locally:

```bash
# To test applying your new migration cleanly:
./scripts/migration.sh reset
```

The reset command will drop the current database completely and apply all migrations in timestamp order. This validates that the entire migration history is sound.

## Running Migrations Remotely

To apply the migrations to your remote Supabase instance (production/staging):

1. Link your local project to your Supabase project (if not already done):
   `npx supabase link --project-ref <your-project-ref>`
2. Run the push command:
   `./scripts/migration.sh push`

## Best Practices

1. **Immutable Migrations:** Once a migration has been merged into the main branch or deployed to an environment, **never modify it**. If you made a mistake, create a new migration to fix it.
2. **Small, Atomic Changes:** Break down large schema changes into smaller, logical migrations.
3. **Data Migrations:** Be careful when writing migrations that mutate data (e.g., splitting a column). Ensure your query handles batches efficiently and test the operation against staging.
4. **Test the Rollback:** Before submitting a critical migration, test its rollback procedure.
